Python并发编程（三）——乐观锁与悲观锁

HeroHR 发表于 2017-11-27    标签： server / python

悲观锁

互斥锁，可重入锁，自旋锁等都属于悲观锁。

当你读取或者写入数据时，你都悲观地认为有别人正在改动数据，所以你希望在你 操作数据的时候上锁，防止别人改动。当然，当已经有别人上锁，你必须等待别人操作完毕。

那么，由此而来的等待时间，在某些场景下影响了性能。

乐观锁

乐观锁以版本号作为分辨更新有效的手段。

当你读取数据的时候，同时读取一个版本号，当你准备以刚刚读取的数据为基础来更新数据时，需要对比刚刚的版本号与现在的版本号是否一致，如果一致，则版本号自增，如果不一致，说明有其他人已经修改过数据了，那么你这次的修改就将放弃。

乐观锁避免了“长事务”下读写的性能问题。

在并发程序中，多个线程同时操作一个资源的时候，有可能发生不可预料的异常。

此时，需要一个能够让两个线程相互了解对方的状态，并在对方操作某些资源把自己阻塞，等待对方操作完毕。

threading.Lock 对象 قفل

这个对象只有两种方法，lock.acquire(), lock.release()

Lock是一把锁子，这个锁不对别的资源加锁，而是对自己加锁。Lock对象加锁的时候，其他地方如果尝试加锁，则会阻塞（等待）直到Lock对象解锁。

当操作完公共资源而忘记解锁时，则会出现死锁情况，这时候将会一直阻塞，一定要避免死锁的出现。

Python并发编程（一）——线程池

HeroHR 发表于 2017-10-16    标签： server / python

Python并发库concurrent.futures

Python的多线程与多进程库threading和multiprocess库提供了简单的并发封装。但是我们在高并发的情况下，往往需要限制线程数和进程数。concurrent.futures库提供了非常好用的线程进程池，我认为是可以代替threading库和multiprocess库了！

基类 Executor

线程池和进程池都继承于这个类，所以提供了一套通用的方法去提交futures。

submit(fn, *args, **kwargs)

这个方法必须接受一个函数为参数，并在后面接收该函数的参数。如果有一个sum(a, b, float=True）的函数需要异步执行，那么只需要executor.submit(sum, a, b, float=True)如此调用，即可将此异步事件放入线程池执行。

这个方法返回一个 future对象。

map(fn, *iterables)

这个方法必须接受一个函数为参数，然后将后面的可迭代参数内容进行打包，变成函数的参数。如果有5个sum函数需要放入线程池执行，那么map(sum, (1, 2, 3, 4,), (5, 6, 7, 8))即等于submit(sum, 1, 5), submit(sum, 2, 5)….四个submit。

这个方法返回一个future列表。

shutdown()

关闭线程池。当使用上下文管理器时，不用显式调用这个方法，上下文管理器会自己关闭线程池。

线程池的两种创建方式

executor = ThreadPoolExecutor(max_workers=x)

这个函数可以创建一个线程池，最大线程数等于x。

with ThreadPoolexecutor(max_workers=x):

这个上下文管理器可以当你不用的时候调用shutdown()方法，关掉线程池。

Future 类

上文提到了submit方法会返回一个future对象，future对象是Future类的实例。通过future对象，我们可以知道我们的异步事件执行到何种程度，可以拿到函数的返回值。

cancle()

取消调用，如果已经执行中且无法终止，会返回False，如果可以取消则返回True。

cancelled()

查看是否被取消。

running()

查看是否正在运行。

done()

查看是否完成。

result（timeout=None）

这个方法比较重要，返回函数的调用结果。当没有设置timeout时，将一直等到函数执行完毕，无论这个函数有没有被执行。当设置timeout之后，超时则会引发timeout错误。

exception(timeout=None)

获取函数的错误。

add_down_callback(fn)

给future添加回调函数，回调函数应该仅接受一个参数 future，当future完成后，将会调用这个回调函数，并将future对象传入。

as_complete(future_list, timeout=None)函数

模块函数，接受future列表。是一个生成器，每当有一个future完成，将生成这个future对象。一般是用做for循环的迭代器的。

一旦futures里有future完成，将会进入一次循环，并打印result值。

最近开了一个新的idea，主要功能是仿照Fabric进行自动化部署与运维。
每有些人写代码是自下而上的，有些人写代码是自上而下的。
在前者的coding中，总会出现把细节处理结束后不知道怎么耦合各个模块。

尝试使用try…except…else流程控制

如果你写过Go的话，你就知道有多么痛苦。Python有异常机制，能够让程序捕获异常，还有那么好的捕获异常关键字。在一个复杂的流程中，可能出现各种不可预料的异常。当发生某项异常之后，你可以选择使用像Go一样层层return，也可以选择直接抛出异常，结束函数，让外部处理这个异常。
用这样的异常控制的语句，会明显清晰代码逻辑，避免繁杂的if判断语句。

尝试使用包来进行去耦合

当一个项目需要用到很多的需要轮子需要自己造的时候，不妨使用包来把轮子写进去，在包文件的__init__.py文件里导入包中的异常类以及各种接口和工厂函数，然后让框架去导入包文件，比在同层文件夹写许多py文件要舒服的多。

尝试用pass忽略细节

我爱pass关键字！
当某些你在一气呵成的写某个逻辑的时候，突然发现需要考虑另一种情况，这时候你当然是选择继续写下去，忽略掉另一种情况。写个else分支，把它就那样放着，或者加个字符串"""Need to fix"""，不如把字符串换成pass关键字，当你回顾你的代码时候发现了pass时，你就会知道，你又得要继续码代码了！

尝试使用基类和duck type

当代码中出现了很多耦合的地方，比如某类拥有同样的方法，甚至两个类仅仅是名字不同…
考虑使用基类把他们的共同点写入一个类中，然后再从中继承。
或者使用一个接口把方法名相同，返回值意义相同的类抽象。
不说让别人看了，起码自己看起来舒服，让IDE看起来舒服。
