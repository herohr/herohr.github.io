深入理解HTTP——确定HTTP报文实体长度

HeroHR 发表于 2017-05-25    标签： server / HTTP

如果将HTTP是一列火车，那么实体或称正文就是火车的一大串火车皮。

在HTTP1.0的情况下

HTTP1.0是一个较为低级的版本，现代浏览器和服务器已经弃用。但1.0版HTTP却是HTTP协议无状态的最好的体现。在HTTP1.0中，一次请求及其响应对应一个TCP连接。
也就是说，这列火车就拉一节火车皮！
TCP连接建立起来之后，客户端就开始传输首部，然后以\r\n\r\n来标志首部的结束和实体的开始（当然是请求里包含实体才会有实体的开始），接下来就是实体的传输，当实体传输完之后，客户端就开始接收数据，服务器就知道，这次请求就已经结束了，那么实体就是\r\n\r\n到停止接收的那么一段数据。对应的，客户端接收响应的时候也是这样。
因为TCP的三次握手，以及TCP慢启动拥塞，导致这种传输方式效率极为低下，但是模型却很好理解！
改善HTTP1.0的效率的办法只有一个，就是并行请求。这样对客户端来说，的确可能会增加效率，但是对服务器来说，一个客户端对应的几十个TCP长连接，肯定吃不消。

可爱的HTTP1.1

HTTP1.1是一个使用较为广泛的版本，一般浏览器和服务器都遵守HTTP1.1。HTTP1.1的特性就是可以进行持久连接keep-alive。

keep-alive

当首部包含Connection: keep-alive的时候，这个连接便是一个持久连接。
一个持久连接意味着，这个TCP连接是可复用的。当一次HTTP请求结束之后，又可以紧接着使用这个TCP通道发送其他的HTTP请求，这样就减少了TCP握手时间、慢启动，减轻了服务器的压力。

那么问题来了，一个TCP连接里面会有这么多报文传输，确定第一个请求的首部是个很简单的事情，怎么确定他的实体长度，以及怎么分割这个HTTP请求实体后面的请求呢？

Content-Length

当首部包含Content-Length: <length>时，我们便可以确认这条HTTP实体的长度了！
Content-Length首部指的是实体的长度且是实体的真实长度！无论实体是经过怎样的压缩，编码，跑在TCP栈之上的长度就是这么长。那么我们将请求头之后的Content-Length字节长度的数据作为实体，就可以确定第一个HTTP的数据报内容。依次，我们可以确定TCP连接里面的所有HTTP报文。

Chunked

并不是所有持久连接都会包含Content-Length首部的，实体还可以分块传输。
当首部包含Transfer-Encoding: chunked时，这段报文的实体就是分块传输的。分块传输并不是分次传输，也就是传输实体的时候不包含首部，仅仅传输分块的实体。那么怎么确定一块数据的长度呢？
对于每一块数据来说，都是以一个十六进制的数作为开头的然后以\r\n分割长度与实体内容。这段十六进制的数就代表的是接下来一块数据的长度。
我们确定了第一块的长度，就可以确定所有块的长度。
对于分块传输，标志传输结束的是一个以0作为开头的然后一\r\n\r\n作为结尾的的一个0长度数据块。当接收到这个数据块的时候，就意味分块传输已经结束啦！
分块传输，在Web流媒体应用中非常重要，服务器不断的生成数据，客户端不断的接收数据。

展望HTTP2.0

HTTP1.1中，每一个HTTP数据报的传输在TCP栈上依旧是串行的。也就是说，你看电影用的是分块传输，但是如果要发一条弹幕，还得再开一个TCP连接。
在很多情况下，TCP栈上的数据传输并不是满负荷的，明明可以在传输的间隙传输别的数据，但是就是得等一条HTTP报文的结束。这种传输，说到底和HTTP1.0一样都需要并行来解决效率。
HTTP2.0中，每一个HTTP数据包的传输在TCP栈上是并行的。也就是说，在看电影的同时，这条TCP通道还能发送你的弹幕。那么就不用瞬间开那么多的TCP连接了！喜大普奔！
问题接踵而至，如果并行，如何确定每条数据属于那个HTTP报文？

深入理解HTTP——探索HTTP请求的完整路径

HeroHR 发表于 2017-05-20    标签： server / HTTP

一次HTTP请求

从浏览器发起对某URL的HTTP请求开始，到浏览器接收到服务器响应为止，探索一次HTTP请求的完整路径

构造请求

请求头是由浏览器构建的，浏览器会根据自身的情况设置请求头的HTTP版本、接受Gzip压缩、自身浏览器版本、移动端或桌面端等构造请求头，然后根据用户的输入构建请求主题，打包成一个Request，准备发送。

URL解析

如果URL是带有域名的，必须要把域名解析成IP地址。
浏览器从自身缓存里面寻找对应域名的IP地址，如果没有，那么前往系统的解析缓存里面去寻找，如果继续没有，那么就会使用DNS服务来获得域名的IP地址。

发起请求

获取到服务器的IP之后，根据URL的地址与服务器建立TCP连接。
这里需要注意TCP的复用：
如果建立了Keep-Alive连接，那么TCP连接将不会断掉，以供浏览器再次使用这个TCP连接发送数据报文。
如果建立了管道化连接，那么TCP连接将变成全双工的管道，浏览器在使用一个TCP连接发送数据报的同时会接受这个TCP连接由服务器端发送的数据报。
如果建立的是串行连接，那么将会根据IP和端口与服务器进行三次握手建立TCP连接，然后再用这个TCP连接发送数据。

接受请求

在服务器上，这个请求可能被Nginx与Apache服务器首先接收，Nginx或Apache服务器对请求头进行分析，判断这个请求是否合法，然后再次对这个请求进行修饰，建立与后端应用服务器的连接，将请求转发过去。
Nginx这类服务器的作用会把不合法的请求就给拦截掉、在完整接收到数据报之后才会转发后端、对请求进行分发负载均衡，总之起到修饰的作用。

处理请求

应用服务器收到这条请求之后，开始进行业务处理

返回响应

应用服务器构造一个响应，转发给Nginx或Apache服务器，Nginx或Apache再将响应传给客户端。

接收响应

浏览器接收响应，根据建立的TCP连接类型决定是否关闭TCP连接。然后浏览器进行解析与处理，对于重定向，继续访问新的URL，对于响应中引用的其他HTTP连接，浏览器继续请求，然后呈现给用户一个完整的HTML界面，一次请求就这样结束了。

深入理解HTTP——HTTP报文详解

HeroHR 发表于 2017-05-18    标签： server / HTTP

何为HTTP报文

HTTP是一个“客户端-服务器”的协议，由任意端发起的、发送的数据 称为HTTP报文

报文的组成

HTTP报文是文本组成的，是人类可读的字符。
HTTP报文的的请求头必须以回车换行符（\r\n）来区分每一行，请求头与正文必须也有一个\r\n作为分隔。

请求报文

起始行

起始行是TCP建立之后发送的第一行数据，包含请求方法、请求的资源地址、客户端的HTTP版本。

一个简单的HTTP请求报文起始行如下：
GET / HTTP/1.1\r\n
GET方法是一个最简单的HTTP请求方法，代表着获取服务器上的某个资源。
/是服务器的URL资源地址，如果请求的网址是www.baidu.com那么/就是直接访问这个网站的首页
HTTP/1.1是最广泛的HTTP版本，现行浏览器都支持。
最后是一个回车换行符，代表着请求报文的起始行已经结束。

请求头

请求头应该包括起始行，这里主要指除了起始行的内容。
请求头一般包含一些简单的键值对
Key: Value\r\n
可以有多个键值对，都以回车换行符分隔。

请求头一般作为请求的解释，比如告诉服务器接不接受gzip压缩，可以支持那些类型的数据，有些API的设计中在请求头中进行鉴权。
最后一个后面应该有两个回车换行符，作为请求头部分和正文部分的分隔。

响应报文

起始行

响应报文的起始行由HTTP版本号、响应状态码、响应短语组成。
版本号与请求报文的语法一致，不再赘述。
响应状态码与响应短语相对应，状态码是三位整数组成的一个数字，响应短语是一段字符串。
如最常见的响应200 OK,404 NOT FOUND就是响应状态码和响应短语。
状态码为200~299的代表为成功状态码，说明你的请求是合法的并且服务器作出了相应的正确的响应，典型的有200 OK 201 Created（请求的对象已经创建）。
状态码为300~399的代表为重定向状态码，说明你请求的资源已经被移到了其他的地方，典型的有302 Found，浏览器接收到这样的响应就会重定向到相应的URL。
状态码为400~499的为客户端错误的状态码，说明你的请求是不合法的，服务器拒绝了你的请求。典型的有404 Not Found说明你访问的资源已经没有了、401 Unauthorized 说明你的请求未经过认证。
状态码为500~599的为服务器错误状态吗，说明你的请求无论合不合法的但是服务器在处理的时候出了一些错误，导致无法给予正确的响应。典型的有500 Internal Server Error说明服务器出错了、502 Bad Gateway说明代理或者网关出了一些问题。

请求头

响应报文的请求头语法与请求报文的请求头语法一致。
注意一些302重定向的首部，如果服务器给出了302重定向，那么会在请求头里包含一行键为Location，值为重定向URL，浏览器会继续访问Location里的URL。
